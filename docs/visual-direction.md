# Visual Direction

fetchtype should not look like another AI landing page full of soft gradients and six equal-sized rectangles. The product is closer to infrastructure, validation, and signal processing. The visual system should communicate precision, momentum, and policy enforcement.

## Primary Rule

- Avoid the default 2x3 bento-grid layout.
- Avoid evenly weighted cards with identical padding, border radius, and hierarchy.
- Avoid generic dashboard wallpaper that could belong to any SaaS tool.

## Icon System

Use a small family of named icon primitives instead of random feature illustrations.

- `Gate`: policy enforcement, CI checks, rule locks
- `Pulse`: validation events, alerts, threshold breaches
- `Orbit`: themes, modes, and brand variants
- `Beam`: delivery, build output, distribution
- `Weave`: token relationships, aliases, typography systems
- `Lens`: inspection, diagnostics, review workflows

The icon style should be:

- Monoline or near-monoline
- Open shapes instead of filled blobs
- Slightly technical, with cut corners or offset anchors
- Built on repeated geometry so the set feels like one system
- Sparse with color, relying on one accent at a time

## Layout Systems

Use one of these structures instead of equal card grids:

- Signal rail: one dominant left-to-right narrative with checkpoints, annotations, and outcomes
- Orbit map: central product claim with surrounding capabilities connected as satellites
- Split proof: one oversized proof artifact on one side, compressed supporting evidence on the other
- Command slab: terminal or code output as the anchor, with commentary wrapped around it
- Compliance arc: rules and diagnostics arranged as a flow from authoring to enforcement to release

## Information Presentation

When presenting features, avoid listing six peers with identical weight. Use hierarchy.

- Show one big proof, two supporting arguments, and one call to action
- Turn validation output into annotated excerpts, not generic stat tiles
- Use comparison strips, checkpoints, and timelines before you use cards
- Let diagrams breathe; asymmetry is a feature, not a bug

## Launch Surface Guidance

For the first marketing page or dashboard:

- Start with a command-first hero anchored by a custom icon, not an abstract gradient block
- Show a validation or PR-review artifact immediately so the value is concrete
- Use typography scale aggressively; oversized type is better than extra boxes
- Reserve boxed modules for true data groupings, not every piece of content

## Prompting Guidance For Future Agents

When asking for UI or marketing work in this repo, be explicit:

- Say `icon-led`, `asymmetric`, `command-line anchored`, or `diagram-first`
- Say `avoid bento cards`, `avoid equal feature tiles`, and `avoid generic AI SaaS patterns`
- Ask for one dominant visual proof element and a non-grid supporting structure

This should be the baseline visual constraint until the repo has a real design system or launch site implementation.
