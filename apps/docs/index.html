Install

pnpm add -D fetchtype

Or with npm:

npm install -D fetchtype

Commands

fetchtype init

Writes a starter token file to fetchtype.tokens.json. The file includes typography definitions for 12 contexts (heading, body, button, caption, label, input, code, blockquote, and more), light and dark color palettes, a spacing scale, layout breakpoints, and heading hierarchy with two named themes. It passes all validation checks out of the box. 

pnpm exec fetchtype init
pnpm exec fetchtype init --force              # overwrite existing
pnpm exec fetchtype init custom.json          # custom path
pnpm exec fetchtype init --preset dashboard   # from a preset
pnpm exec fetchtype init --prompt "modern SaaS dashboard"  # natural language

--preset accepts: base, editorial, dashboard, ecommerce, docs. --prompt generates a token file from a natural-language description.

fetchtype validate

Checks a token file against 13 rules for schema correctness, accessibility violations, and structural problems. Exits with code 1 if any errors are found, so it works directly as a CI gate. 

pnpm exec fetchtype validate -i fetchtype.tokens.json
pnpm exec fetchtype validate -i tokens.json --json      # machine-readable
pnpm exec fetchtype validate -i tokens.json --github    # PR annotations

fetchtype build

Runs validation first (stops if it fails), then exports to the specified format. Supports CSS custom properties, Tailwind config partials, shadcn CSS with HSL variables, and W3C Design Tokens format. 

pnpm exec fetchtype build -i fetchtype.tokens.json
pnpm exec fetchtype build -i tokens.json --format tailwind
pnpm exec fetchtype build -i tokens.json --format shadcn
pnpm exec fetchtype build -i tokens.json --format w3c
pnpm exec fetchtype build -i tokens.json --report          # standalone HTML report
pnpm exec fetchtype build -i tokens.json --prefix ds       # --ds-typography-*

fetchtype preview

Starts a local dev server that renders your token file with live reload. Shows heading hierarchy, body text, buttons, color palettes with contrast ratios, spacing scale, and a mode switcher. 

pnpm exec fetchtype preview -i tokens.json

fetchtype suggest

Recommends fonts from a catalog of 31 Google Fonts, filtered by typographic context. Returns font families suited for the specified role with metadata on style, weight range, and classification. 

pnpm exec fetchtype suggest --context display
pnpm exec fetchtype suggest --context interface
pnpm exec fetchtype suggest --context reading
pnpm exec fetchtype suggest --context mono

fetchtype import

Imports a W3C Design Tokens file and converts it to fetchtype token format. 

pnpm exec fetchtype import -i w3c-tokens.json

fetchtype mcp

Starts an MCP (Model Context Protocol) server over stdio with 5 tools: validate, build, suggest, init, and presets. Enables AI agents to interact with fetchtype programmatically. 

pnpm exec fetchtype mcp

Output formats

--format accepts: all (default), css, json, tailwind, shadcn, w3c. --prefix sets the CSS variable namespace (default: ft). --out-dir controls where files land (default: dist/tokens). --json on validate gives machine-readable diagnostic output. --report on build generates a standalone HTML report.

Token file format

A fetchtype.tokens.json file has six sections: typography, color, spacing, layout, hierarchy, and optionally themes and modes. All are required except themes and modes. 

{
  "typography": {
    "heading":    { "fontFamily": [...], "fontSize": "3.052rem", "fontWeight": 700, "lineHeight": 1.1 },
    "body":       { "fontFamily": [...], "fontSize": "1rem",     "fontWeight": 400, "lineHeight": 1.6 },
    "button":     { "fontSize": "0.9375rem", ... },
    "caption":    { ... },
    "label":      { ... },
    "input":      { ... },
    "code":       { ... },
    "blockquote": { ... },
    "subheading": { ... },
    "chart-label": { ... },
    "chart-axis":  { ... },
    "chart-title": { ... }
  },

  "color": {
    "light": {
      "text":        { "primary": { "value": "#111827" }, "secondary": {...}, "muted": {...} },
      "background":  { "primary": { "value": "#ffffff" }, "secondary": {...} },
      "border":      { "default": {...}, "muted": {...}, "accent": {...} },
      "interactive": { "default": {...}, "hover": {...}, "active": {...}, "disabled": {...} }
    },
    "dark": { /* same structure */ }
  },

  "spacing": {
    "unit": "0.25rem",
    "scale": { "xs": "0.25rem", "sm": "0.5rem", "md": "0.75rem", "lg": "1rem", ... }
  },

  "layout": {
    "maxWidth":     { "prose": "68ch", "content": "72rem", "wide": "90rem", "full": "100%" },
    "breakpoints":  { "sm": "40rem", "md": "48rem", "lg": "64rem", "xl": "80rem" },
    "grid":         { "columns": 12, "gap": "1.5rem" }
  },

  "hierarchy": {
    "scale": "major-third",
    "baseSize": "1rem",
    "headings": {
      "h1": { "fontSize": "3.052rem", "lineHeight": 1.08 },
      "h2": { "fontSize": "2.441rem", "lineHeight": 1.1 },
      ...
    }
  },

  "themes": [
    {
      "name": "brand-ocean",
      "colorScheme": "brand",
      "selector": "[data-theme=\"brand-ocean\"]",
      "tokens": { "color.light.interactive.default.value": "#0f766e" }
    }
  ],

  "modes": {
    "display": {
      "name": "display",
      "tokens": { "typography.heading.lineHeight": 0.95, "typography.heading.letterSpacing": "-0.02em" }
    },
    "interface": {
      "name": "interface",
      "tokens": { "typography.body.fontSize": "0.875rem", "typography.body.lineHeight": 1.45 }
    },
    "reading": {
      "name": "reading",
      "tokens": { "typography.body.lineHeight": 1.7, "typography.body.fontSize": "1.125rem" }
    },
    "mono": {
      "name": "mono",
      "tokens": { "typography.body.fontFamily": ["JetBrains Mono", "monospace"] }
    }
  }
}

Token references

Values can reference other tokens with {path.to.value} syntax. References are resolved at build time. Circular references are detected and reported as errors. 

"accent": { "value": "{color.light.interactive.default.value}" }

Typography contexts

The typography section supports 12 named contexts: heading, subheading, body, caption, button, label, input, code, blockquote, chart-label, chart-axis, and chart-title. Body and button are required — validation errors if either is missing. Each context accepts fontFamily (string array), fontSize, fontWeight, lineHeight, and letterSpacing.

Themes

Named themes override any token path. Each theme gets its own CSS block using the selector you specify (default: [data-theme="name"]). Theme names must be unique. Themes can override typography, color, spacing, layout, and hierarchy values.

Presets

Presets provide opinionated starter token files tuned for specific use cases. Each preset passes all 13 validation rules, includes WCAG-safe contrast ratios for both light and dark themes, and ships with appropriate type scales, spacing, and color palettes. 

pnpm exec fetchtype init --preset editorial

Available presets

  • base — general-purpose defaults. Major-third scale, system font stack, neutral palette with light and dark color tokens. The default when no preset is specified.
  • editorial — long-form reading. Larger body text (1.125rem), generous line-height (1.7), serif headings, narrower prose width (65ch).
  • dashboard — data-dense interfaces. Compact body text (0.875rem), tighter line-height (1.5), monospace for data, wider content area.
  • ecommerce — product and conversion pages. Clear heading hierarchy, prominent button sizing, high-contrast interactive states in both light and dark themes.
  • docs — documentation sites. Code-friendly with monospace support, balanced heading scale, optimized for scanning and navigation.

Modes

Typography modes override specific tokens for a given rendering context. They are optional — a token file works without them. When present, each mode is validated independently and emits its own CSS block.

Built-in modes

  • display — tight heading line-height (0.95), negative letter-spacing. For hero sections and large type.
  • interface — smaller body text (0.875rem), compact line-height (1.45). For UI-dense screens like dashboards and settings.
  • reading — larger body text (1.125rem), generous line-height (1.7). For long-form content.
  • mono — monospace font family. For code, terminal output, data tables.

CSS output

Each mode generates a [data-mode] CSS block: 

[data-mode="display"] {
  --ft-typography-heading-line-height: 0.95;
  --ft-typography-heading-letter-spacing: -0.02em;
}

[data-mode="reading"] {
  --ft-typography-body-line-height: 1.7;
  --ft-typography-body-font-size: 1.125rem;
}

Composing modes with themes

Modes and themes are orthogonal. A page can use [data-theme="dark"] and [data-mode="reading"] simultaneously. Modes override typography tokens; themes override color tokens (including dark mode contrast pairs). Both compose without conflict — and fetchtype validates every combination.

What gets checked

Validation runs 13 rules across both light and dark themes: schema checks first (hard errors), then accessibility (WCAG contrast for each theme), readability, structural, and font-loading checks.

Schema (errors)

  • All required fields present and correctly typed
  • Typography context names from the allowed enum
  • Color structure: light and dark modes with text, background, border, interactive groups
  • Layout: maxWidth, breakpoints, optional grid
  • Hierarchy: valid scale enum, heading levels h1-h6
  • Themes: unique names, valid token override paths

Accessibility (errors)

  • Contrast ratio ≥ 4.5:1 (WCAG AA) — primary text against primary background, validated separately for light and dark themes. Each reports its own ratio (contrast.light, contrast.dark) using WCAG 2.1 relative luminance (0.2126R + 0.7152G + 0.0722B). Ratios ≥ 7:1 are flagged as WCAG AAA.
  • Required contexts — body and button typography must be defined. These are the minimum for any functional interface.

Readability (warnings)

  • Body line-height ≥ 1.5 — WCAG 1.4.12 requires spacing of at least 1.5x the font size for body text.
  • Button font-size ≥ 14px — interactive text needs to be large enough to read and tap reliably on touch devices.
  • Caption/label font-size ≥ 11px — small text contexts must remain legible. Below 11px, text becomes unreadable at standard screen densities.
  • Prose max-width ≤ 75ch — lines longer than 75 characters increase reading fatigue and reduce comprehension.

Structural (warnings)

  • Heading sizes decrease h1 → h6 — heading font sizes must form a strictly decreasing sequence from h1 through h6.
  • Heading line-height < body line-height — headings use tighter leading than body text. If heading line-height exceeds body line-height, the visual hierarchy breaks.
  • Spacing scale monotonically increasing — spacing values (xs, sm, md, lg, xl, etc.) must increase from smallest to largest without inversions.
  • Scale divergence within ±10% — computed heading sizes from the named ratio should not diverge more than 10% from the declared heading sizes.
  • Dark mode completeness — if a light theme defines a color token, the corresponding dark theme must also define it. Missing dark-mode values are flagged so dark themes never ship with untested contrast or missing overrides.

Font loading (warnings)

  • Font fallback chains present — every fontFamily array must include at least one generic fallback (sans-serif, serif, monospace, etc.).
  • Font payload estimate ≤ 150KB — estimated total font payload across all declared families. Exceeding 150KB triggers a warning with the estimated size.

References (errors)

  • All {path.to.token} aliases resolve to actual values
  • No circular reference chains
  • Reference paths must point to valid locations in the token tree

Exporters

fetchtype build writes validated tokens to the format your stack needs. Each exporter produces files in the output directory (default: dist/tokens).

Tailwind

pnpm exec fetchtype build -i tokens.json --format tailwind

Generates a tailwind.config.ts partial with your typography, colors, and spacing mapped to Tailwind's theme structure. Drop it into your Tailwind config's theme.extend. 

// tailwind.config.ts (generated partial)
export default {
  theme: {
    extend: {
      fontSize: {
        heading: "3.052rem",
        body: "1rem",
        button: "0.9375rem",
      },
      lineHeight: {
        heading: "1.1",
        body: "1.6",
      },
      colors: {
        text: { primary: "#111827", secondary: "#374151" },
        bg: { primary: "#ffffff", secondary: "#f9fafb" },
      },
    },
  },
};

shadcn

pnpm exec fetchtype build -i tokens.json --format shadcn

Generates CSS with HSL variables matching shadcn/ui conventions. Compatible with shadcn's theming system — use alongside your existing shadcn config. 

:root {
  --background: 0 0% 100%;
  --foreground: 220 13% 9%;
  --primary: 220 13% 9%;
  --primary-foreground: 0 0% 100%;
  /* ... */
}

.dark {
  --background: 220 13% 9%;
  --foreground: 210 20% 98%;
  /* ... */
}

W3C Design Tokens

pnpm exec fetchtype build -i tokens.json --format w3c

Exports to the W3C Design Tokens Community Group format. Useful for interop with other design token tools (Style Dictionary, Tokens Studio, etc.). 

pnpm exec fetchtype import -i w3c-tokens.json

Import works in the other direction: convert a W3C token file into fetchtype format.

Preview and report

Live preview

pnpm exec fetchtype preview -i tokens.json

Starts a local dev server that renders your token file. The preview includes heading hierarchy, body text samples, button states, color palettes with computed contrast ratios, spacing scale visualization, and a mode switcher. Changes to the token file trigger a live reload.

HTML report

pnpm exec fetchtype build -i tokens.json --report

Generates a standalone HTML report alongside the normal build output. The report contains the same views as the preview server: heading hierarchy, body text, buttons, color palettes with contrast ratios, spacing scale, and mode switcher. Use it as a CI artifact or share it with designers for review.

Font intelligence

Font suggestions

pnpm exec fetchtype suggest --context display
pnpm exec fetchtype suggest --context interface
pnpm exec fetchtype suggest --context reading
pnpm exec fetchtype suggest --context mono

Recommends fonts from a curated catalog of 31 Google Fonts. Each suggestion is filtered by the specified context — display fonts optimize for impact at large sizes, interface fonts for legibility at small sizes, reading fonts for sustained comprehension, and mono fonts for code and data.

Font validation

Three font-related rules run during validation:

  • Missing fallbacks — every fontFamily array must end with a generic family (sans-serif, serif, monospace, system-ui, cursive).
  • Payload estimate — estimated total font weight across all declared families. Warns if the estimate exceeds 150KB.
  • Unknown font family — flags font families not found in the catalog (informational, not a blocking error).

Agent integration

fetchtype is an agent-native typography stack. AI coding agents can validate tokens, generate configurations, and get font recommendations through the MCP server or the natural-language init command.

MCP server

pnpm exec fetchtype mcp

Starts an MCP (Model Context Protocol) server over stdio. The server exposes 5 tools:

  • validate — run validation on a token file, returns diagnostics
  • build — export tokens to any supported format
  • suggest — get font recommendations for a context
  • init — generate a starter token file from a preset or prompt
  • presets — list available presets with descriptions

Any MCP-compatible agent (Claude, Cursor, Windsurf, etc.) can connect to the server and use these tools directly.

Natural-language init

pnpm exec fetchtype init --prompt "modern SaaS dashboard with dark mode"

Generates a token file from a natural-language description. The output passes all 13 validation rules and can be further refined by hand or by an agent.

GitHub CI

Add --github to any validate command to get two things: inline annotations on the PR diff, and a step summary with the full diagnostic report.

Workflow file

# .github/workflows/typography.yml
name: Typography
on: [pull_request]

jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v2
      - run: pnpm install
      - run: pnpm exec fetchtype validate -i fetchtype.tokens.json --github

What --github emits

  • Workflow annotations — ::error, ::warning, and ::notice commands that appear inline on the PR diff. Each annotation includes the rule name, the source file, and the diagnostic message with actual values.
  • Step summary — a markdown report appended to GITHUB_STEP_SUMMARY with pass/fail status, source file, error/warning/info counts, and a bulleted list of every diagnostic.

Build output

fetchtype build produces files in the output directory (default: dist/tokens). The exact files depend on the chosen format.

tokens.css

:root {
  --ft-typography-heading-font-family: Inter, system-ui, sans-serif;
  --ft-typography-heading-font-size: 3.052rem;
  --ft-typography-heading-font-weight: 700;
  --ft-typography-heading-line-height: 1.1;
  --ft-typography-body-font-size: 1rem;
  --ft-typography-body-line-height: 1.6;
  --ft-color-light-text-primary: #111827;
  --ft-color-light-background-primary: #ffffff;
  --ft-spacing-unit: 0.25rem;
  --ft-spacing-sm: 0.5rem;
  --ft-spacing-lg: 1rem;
  --ft-layout-max-width-prose: 68ch;
  /* ... */
}

[data-theme="dark"] {
  --ft-color-dark-text-primary: #f9fafb;
  --ft-color-dark-background-primary: #111827;
  /* ... */
}

[data-mode="display"] {
  --ft-typography-heading-line-height: 0.95;
  --ft-typography-heading-letter-spacing: -0.02em;
}

[data-mode="reading"] {
  --ft-typography-body-line-height: 1.7;
  --ft-typography-body-font-size: 1.125rem;
}

[data-theme="brand-ocean"] {
  --ft-color-light-interactive-default: #0f766e;
  /* ... */
}

tokens.json

The full resolved token set with all aliases expanded. Same structure as the input file, but every {reference} has been replaced with its concrete value. Useful for feeding into Tailwind configs, Figma plugins, or any tool that consumes structured tokens.

Using the output

The CSS file is a standalone stylesheet. Import it or link it — no build plugin needed. Use the custom properties anywhere: 

body {
  font-size: var(--ft-typography-body-font-size);
  line-height: var(--ft-typography-body-line-height);
  color: var(--ft-color-light-text-primary);
  max-width: var(--ft-layout-max-width-prose);
}
